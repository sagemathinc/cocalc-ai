# Codex App-Server Image Generation Plan

## Goal

Add CoCalc support for images generated by Codex via the upstream app-server `image_generation` Responses tool, without adding a new Codex fork or changing the local Codex build beyond the existing TCP timeout patch.

This should initially be **ChatGPT-subscription-only**. API-key/site-key support is explicitly out of scope for the first version, since upstream Codex only exposes the built-in image generation tool for ChatGPT auth.

## Current State

OpenAI/Codex upstream already has the important backend pieces:

- `codex-rs` has a native Responses API tool spec:
  - `ToolSpec::ImageGeneration { output_format }`
  - serialized as `{ "type": "image_generation", "output_format": "png" }`
- The tool is gated by:
  - `Feature::ImageGeneration`
  - ChatGPT auth mode
  - a model that supports image modality
- `codex-rs` handles `image_generation_call` response items.
- Generated image payloads are saved to:
  - `${CODEX_HOME}/generated_images/<session-id>/<image-call-id>.png`
- App-server protocol has `ThreadItem::ImageGeneration`.
- TUI already renders generated image history cells.

CoCalc currently uses Codex app-server, not the Codex TypeScript SDK, so the SDK's incomplete image-generation typing is not a blocker.

The local build script `src/scripts/build-local-codex-binaries.sh` currently builds `rust-v0.118.0` with a TCP timeout patch. That tag already contains the image-generation plumbing, so this plan does not require another Codex patch.

## Product Behavior

When the user asks Codex to generate an image:

1. Codex exposes the upstream `image_generation` tool when the user's auth/model allows it.
2. The model calls the hosted image tool.
3. Codex app-server emits an `imageGeneration` item.
4. CoCalc records the generated image in the Codex activity log.
5. The UI shows a clear "Generated image" row with:
   - revised prompt, if present
   - status
   - saved project file path, if available
   - a preview or link
   - later, a CoCalc blob URL when promotion succeeds

If image generation is unavailable because the turn is using API-key auth or an unsupported model, CoCalc does not need to special-case it initially. Codex simply will not expose the tool.

## Important Constraint

Do not implement a separate CoCalc image-generation API path in phase 1.

Reasons:

- The app-server path preserves Codex's normal model/tool semantics.
- The generated image stays in the Codex turn history as an `image_generation_call`, so future turns can refer to it.
- Billing/entitlement remains aligned with the user's ChatGPT subscription.
- It avoids designing CoCalc's own image-generation quota and cost model before we know how users use this.

## Phase 1: Surface App-Server Image Items

### Backend ACP Event Type

Add an ACP activity event to `src/packages/conat/ai/acp/types.ts`.

Suggested shape:

```ts
{
  type: "image";
  id?: string;
  status: string;
  revisedPrompt?: string;
  savedPath?: string;
  blob?: {
    uuid: string;
    filename: string;
    url: string;
  };
}
```

Notes:

- `savedPath` is the path reported by Codex app-server.
- `blob` is optional and should not block the initial activity event.
- Keep the event small. Do not put base64 image bytes in the activity stream.

### App-Server Item Mapping

Update `src/packages/ai/acp/codex-app-server.ts`.

In `handleItem`, add a case for app-server `imageGeneration` items:

```ts
case "imageGeneration":
  await stream({
    type: "event",
    event: {
      type: "image",
      id: item.id,
      status: item.status,
      revisedPrompt: item.revisedPrompt,
      savedPath: item.savedPath,
    },
  });
  break;
```

The exact app-server field names should be confirmed from a live event or the generated protocol. The upstream generated Python protocol uses:

- `id`
- `status`
- `revisedPrompt`
- `result`
- `savedPath`

The raw `result` is base64 image data. CoCalc should not persist it in the activity log.

### Duplicate Handling

The app-server can emit item updates and item completions. Avoid duplicate image rows by tracking image item ids in the current turn:

```ts
const emittedImages = new Set<string>();
```

Emit only once when:

- `item.type === "imageGeneration"`
- `item.status === "completed"` or `item.savedPath` is present
- `item.id` has not been emitted

If an in-progress event is useful later, add a separate "Generating image..." row, but keep phase 1 simpler: emit on completion.

### Tests

Add focused tests in `src/packages/ai/acp/__tests__/codex-app-server.test.ts`.

Test cases:

- `item/completed` with `type: "imageGeneration"` emits an ACP `image` event.
- the event omits raw `result`.
- duplicate `item/updated` and `item/completed` for the same image id produces one activity event.
- missing `savedPath` still emits a row with status and revised prompt, so failures are visible.

## Phase 2: Frontend Activity Rendering

Update `src/packages/frontend/chat/codex-activity.tsx`.

### Normalize

Add `image` to the activity entry union and map ACP `event.type === "image"` to an image row.

Suggested normalized entry:

```ts
{
  kind: "image";
  id: `image-${seq}`;
  seq;
  time;
  imageId: event.id;
  status: event.status;
  revisedPrompt: event.revisedPrompt;
  savedPath: stringifyPath(event.savedPath);
  blob: event.blob;
}
```

### Render

Render as a compact activity row:

- label: `Generated image`
- status tag: `completed`, `failed`, etc.
- prompt text: muted, truncated when compact
- saved path: clickable using existing activity file-opening behavior
- blob URL: clickable and copyable when available

Initial rendering can be link-only. Preview can be added after the event pipeline is reliable.

### Markdown Export

Update `codexEventsToMarkdown` so generated images export cleanly:

```md
- Generated image: <revised prompt>

  Saved file: `...`
```

If a blob URL exists:

```md
![Generated image](/blobs/generated-image.png?uuid=...)
```

## Phase 3: Promote Generated Images To CoCalc Blobs

This is desirable because blob URLs can be pasted directly into markdown/html/chat and survive outside the local Codex session file path.

There are two viable designs.

### Option A: Backend Promotion During ACP Turn

When `codex-app-server.ts` sees an `imageGeneration` item with `savedPath`:

1. Resolve the path inside the project runtime.
2. Read the PNG bytes.
3. Store bytes in CoCalc's blob store.
4. Emit a second/update activity event with the blob URL.

Pros:

- Works automatically.
- The activity row has a durable markdown-ready URL.
- No browser fetch/upload loop.

Cons:

- Launchpad and Lite blob storage APIs differ.
- Need careful path resolution from Codex project runtime path to the backend process that can read it.
- Need to ensure generated image files are not read from subscription auth cache paths.

### Option B: Frontend Promotion On Demand

Render a "Save as CoCalc blob" action in the image activity row.

When clicked:

1. Browser fetches/reads the generated image through existing project file access.
2. Browser uploads it through existing `/blobs` upload endpoint.
3. Activity row updates or inserts a markdown snippet.

Pros:

- Uses existing frontend `uploadBlobImage`.
- Avoids new backend blob plumbing.
- Easy to make user-controlled.

Cons:

- Requires the generated file to be accessible to the browser.
- Not automatic, so markdown-ready URLs are not immediately available.
- Updating persisted activity with the resulting blob URL needs additional writeback.

### Recommended Blob Plan

Implement backend promotion as a follow-up only after phase 1 and phase 2 are working.

Start with a backend helper behind one interface:

```ts
async function promoteGeneratedImageToBlob(opts: {
  projectId: string;
  savedPath: string;
  filename: string;
}): Promise<{ uuid: string; filename: string; url: string } | undefined>;
```

Then add implementations by environment:

- Lite: use the existing AKV blob store used by `/blobs`.
- Launchpad/project-host: use the existing hub/project blob mechanism or add a project-host RPC if needed.

Do not block the user-visible generated image event on blob promotion. Emit the image row immediately with `savedPath`; append/update blob info when promotion finishes.

## Phase 4: Prompt/UX Guidance

Add a small note somewhere appropriate in Codex help/default guidance:

- Image generation is available when using a ChatGPT subscription and an image-capable model.
- Generated images are saved in the project and shown in the activity log.
- If a CoCalc blob URL is shown, that URL can be pasted into markdown/html/chat.

This should not be prominent until the feature is proven.

## Validation Plan

### Unit Tests

Run focused tests:

```sh
cd src/packages/ai && pnpm test -- codex-app-server.test.ts
cd src/packages/frontend && pnpm test -- chat/__tests__/codex-activity.test.tsx chat/__tests__/codex-activity-markdown.test.ts
pnpm -C src lint:frontend
```

### Manual Test

Using ChatGPT subscription auth:

1. Start a Codex chat turn.
2. Ask: `Generate a small image of a red sailboat on a blue lake.`
3. Confirm Codex activity shows a generated image row.
4. Confirm the saved path exists in the project runtime.
5. Confirm the row survives browser refresh because it is persisted in the activity log.
6. If blob promotion is implemented, confirm the blob URL loads and can be pasted into markdown.

### Negative Tests

- API-key/site-key auth should not expose image generation; no CoCalc error should appear.
- Unsupported model should behave like normal Codex unsupported-tool behavior.
- Missing `savedPath` should show a row with status instead of silently dropping the item.
- Large image payloads must never be persisted in the activity log.

## Risks

### App-Server Field Names

The generated app-server protocol uses camelCase fields, but live events should be confirmed. The implementation should tolerate both `savedPath` and `saved_path` if this is cheap.

### File Path Accessibility

Codex saves generated images under `CODEX_HOME`. In Launchpad subscription mode, auth files are mounted from subscription storage, but project-local `.codex` should still be in project runtime home. Verify this before assuming frontend/project file links work.

### Blob Promotion Scope

Blob promotion is useful, but it introduces environment-specific backend storage concerns. Keep it separate from the first visible image-generation support.

### Model/Feature Entitlement

Do not try to force-enable the tool for API-key/site-key auth in this phase. That would turn this from app-server plumbing into a billing/entitlement feature.

## Implementation Estimate

- Phase 1 backend event support: 1-2 hours.
- Phase 2 frontend activity rendering and markdown export: 2-4 hours.
- Focused tests and manual validation: 1-2 hours.
- Blob promotion: likely 0.5-2 days depending on whether an existing server-side blob write path is reusable in both Lite and Launchpad.

## Recommended First Commit

Implement only:

- ACP `image` event type.
- app-server `imageGeneration` item mapping.
- frontend activity row.
- markdown export.
- unit tests.

Do not implement blob promotion in the first commit. Add a clear saved file link first; then decide whether automatic blob promotion should be backend-first or user-triggered.
